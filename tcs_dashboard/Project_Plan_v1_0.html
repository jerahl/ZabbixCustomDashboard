v1.0 — what this plan covers: The tcs_dashboard Zabbix module already contains six full-page custom dashboards (matching pixel-for-pixel the design bundle from claude.ai/design). The shell, sidebar, topbar, page header, AP navigator, host navigator, port grid, camera wall, RAID grid, and 24h sparkline strips are built and committed on branch claude/update-project-designs-b831Y. The only missing piece is real data — every dashboard reads from a *-data.jsx mock file. This plan covers the wiring work: lift the data-collection layer from three sibling widget repos, write per-dashboard PHP Action*Data JSON endpoints, write per-dashboard *-bridge.jsx adapters parallel to the existing data-bridge.jsx, and ship.
  • Architecture decision (locked): lift data layers, discard widget shells. The widget repos stay installable; their templates + external scripts + PHP API clients become the canonical data sources for the unified module. The existing widget rendering (PHP Widget.php, WidgetForm.php, JS widget.js) does not move — the module's React dashboards replace them.
  • Auth normalisation (locked): Milestone uses {$MILESTONE.HOST} / {$MILESTONE.USER} / {$MILESTONE.PASSWORD} / {$MILESTONE.SCHEME} — the existing macros from jerahl/MilestoneZabbix. M0 audits the XIQ + PacketFence + iDRAC macro names already in use in their respective repos and matches them — no new macro vocabulary invented.
  • Surveillance is priority 1. M2 wires the Surveillance NOC + Camera Detail + Server Detail dashboards first. Switches / Servers / AP-extend run in M3 (parallel tracks possible after M1). Global Dashboard is M4 — it aggregates the others, so it lands last per-dependency, not per-importance.
  • Existing AP Detail wiring stays. ActionDashboard.php + ActionDashboardData.php + data-bridge.jsx already pull live host.get/item.get data for one AP. M3 extends them: inject AP_SITES from a real host.get filtered by host group (currently stubbed), wire the XIQ-side fields by linking to the lifted XIQClient.php, and surface PacketFence client lists from the lifted PF client.
  • Three companion repos get a "see ZabbixCustomDashboard" banner in their READMEs. The user moves them in manually — drafts are at tcs_dashboard/notes/widget-repo-readme-banners.md. The widget repos are not deprecated; they're the canonical home for the templates and API clients while the unified module is the canonical home for the operator UI.
  • Source-of-truth tagging stays. The dashboards already render a SourceBadge primitive next to each metric (ZBX / EXT / PF / IDRAC). Each *-bridge.jsx propagates the badge from the data-controller's response so a cell that says "94°F" always knows whether iDRAC or the OS agent produced it. Per AP Plan v3.2 §B and Milestone Plan §B G16 — operators need to know which probe path produced each value, especially when two paths disagree.
Project context — what exists today, what we're wiring to it: The tcs_dashboard module on branch claude/update-project-designs-b831Y ships six full-page React dashboards rendered via Babel-standalone, all behind one unified sidebar (global-nav.jsx). Five action controllers exist (ActionGlobal, ActionDashboard, ActionSwitches, ActionServers, ActionSurveillance) plus two detail views (ActionCamera, ActionServer). Only ActionDashboard + ActionDashboardData have a JSON refresh endpoint wired to real Zabbix data; the rest render static mock data via nvr-data.jsx / switches-data.jsx / servers-data.jsx / global-data.jsx.
  • Three widget repos hold the data layers we'll lift. jerahl/MilestoneZabbix ships Milestone by HTTP API.yaml + 6 external scripts (camera state, ESS WebSocket dump, refresh) + a milestone_camera_status widget with the PHP/HTTP scaffolding for talking to XProtect's REST API. jerahl/ZabbixSwitchPortWidgets ships EXOS PoE template additions + a PHP-side switch port + MAC learning client + an optional PacketFence-15 client. jerahl/ZabbixExtremeIQ ships XIQClient.php + the XIQ AP template. Each repo's PHP "client" classes are what we want — they encapsulate the auth + HTTP semantics for each external system. The widget rendering layer (Widget.php, WidgetForm.php, widget.js) is what we discard.
  • Module load order matters. The unified sidebar (global-nav.jsx) is loaded first on every page, then per-dashboard data + tabs + app entry. M4 introduces six bridge files that load before their dashboard's *-data.jsx mock and shadow the same window globals. The mocks aren't deleted — they remain as the empty-state fallback when the bridge can't reach its endpoint, so an operator who breaks the network connection to Zabbix sees the demo data, not a white page.
  • Dashboards already poll. data-bridge.jsx polls tcs.dashboard.data every 30 s and re-publishes the window globals. M4's new bridges all match this interval and shape (request → JSON → defensive merge → optional tcs:data CustomEvent). Slow upstreams (XProtect, PF) get coarser intervals at the controller, not the bridge; the bridge always asks every 30 s and the controller serves a cached value if fresh.
  • Cross-repo coordination. The Milestone Dashboard Plan v1.1 (companion plan) is currently in M0/M1 — splitting the Milestone single-host template into a three-host topology (Site / Camera / Recording Server). Our M1 (template imports) consumes whatever shape the Milestone plan ships. If Milestone M1 lands first, our M2 reads camera + RS hosts as first-class Zabbix hosts. If Milestone M1 lags, our M2 falls back to the existing single-Site-host model and reads dependent items by GUID. Both paths produce the same JSON shape on the dashboard side — the bridge doesn't care.
  • Carry-forward gotchas from AP Plan v3.2 §B and Milestone Plan v1.1 §B. AP G16–G29 (Zabbix 7.4 widget-framework requirements) do not apply here — this is a custom module, not a widget. AP G21 (PHP 8.0 not 8.1+) does apply: every Action*Data.php we write conforms to PHP 8.0 idioms. Milestone G1–G15 (camera/RS topology, vendor SNMP, ICMP scale, OAuth2 token shape) flow downstream to us as data-shape constraints, not code constraints. Read both plans before starting M2.

§B — Predicted gotchas + carry-forward C1–C13 module-specific · AP G16–G29 + Milestone G1–G21 inherited
Pre-flight list. Validate C1–C13 in M0; carry-forwards are settled — accept and apply.
C1 — Discard widget shells, lift only data layers. Each source repo is a Zabbix 7.4 widget — Widget.php, WidgetForm.php, widget.js, views/widget.view.php, plus its manifest.json with "type":"widget". None of that moves. The module is a full-page "type":"module" manifest with action controllers; mixing the two would re-trigger AP G16–G29 (widget framework requirements) for no payoff. Resolution: only lib/*Client.php, templates/*.yaml, and externalscripts/*.{sh,py} move. The widget shells stay in their original repos as standalone widgets. Anyone wanting to render a single panel in a native Zabbix dashboard installs the widget repo separately; the unified module is for the full-page view.
C2 — Match each source's existing macro names — don't invent a new vocabulary. Confirmed for Milestone ({$MILESTONE.HOST} / .USER / .PASSWORD / .SCHEME). XIQ + PF + iDRAC need verification in M0 from each source repo's manifest.json + template YAML. Don't ship "we use a new {$TCS_* namespace" — operators already running the source widgets keep their existing macros, and the module reads from the same hosts. Cross-pollinating two macro patterns on one host means engineering for inconsistency forever.
C3 — OAuth2 token fetch: per-call, not cached. The Milestone widget pattern fetches a token inline per HTTP call — no token caching, no refresh dance. MilestoneClient.php in the lifted form keeps this pattern. Tempting to add a 5-minute token cache (Zabbix's CCache or APCu) since the dashboards poll every 30 s, but: (a) the existing template's JS Script items already fetch per-call and at scale that's 300+ calls/min steady-state with no observed problem; (b) introducing a cache layer means a stale-credential edge case where rotated PASSWORDs fail silently for the cache-TTL window. Defer caching to M5 — measure first.
C4 — Babel-standalone first paint cost. The dashboards compile JSX in-browser via Babel-standalone — first-paint ~1.5–2 s on cold cache. Each new bridge adds ~50–200 ms (small files, but Babel still parses each one). At 6 bridges + 4 dashboard JSX files + tweaks/primitives/shell, total compile time on cold cache is ~2.5 s. Acceptable for M2–M4; M5 plans an esbuild build step that produces a single pre-compiled bundle, dropping Babel entirely. Air-gapped installs (already documented in the README) need this anyway because Babel-standalone is the largest CDN dependency.
C5 — Bridges must defensively stub every key consumers read — same pattern as data-bridge.jsx. The mock-data files (nvr-data.jsx, switches-data.jsx, servers-data.jsx, global-data.jsx) define rich shapes — e.g. SWITCH_SITES[i].switches[j].ports[k].poe. The dashboards read these without guarding for missing keys. If a bridge publishes SWITCH_SITES = [], the dashboard renders empty cleanly. If it publishes SWITCH_SITES = [{switches: undefined}], it crashes. Resolution: every bridge has an EXPECTED_KEYS list and an emptyShape() helper that builds a zero-filled but fully-keyed object — same pattern data-bridge.jsx already follows for ZBX_ITEMS. Code-review checklist: every bridge merge needs a "publishes all keys, even on null upstream" sign-off.
C6 — host.get returns hosts visible to the logged-in user — Global Dashboard headlines are per-user. Zabbix's host.get + problem.get respect host-permission filters. An operator with read access to one school's host group sees "12 problems"; the network admin sees "47". This is correct behaviour but different from a server-side dashboard rendered by Zabbix, which most operators are used to and which can be impersonated. Resolution: document in the Global Dashboard help tooltip that counts are scoped to the viewer's permissions; surface a "Showing N of M total hosts" line in the page header when N<M. Don't try to escalate via API token — that's a security regression.
C7 — Cross-host item lookups by tag (consume Milestone Plan G4's foundation). The Milestone Plan v1.1 ships per-camera and per-RS hosts (in M1), with each item tagged cam_id={GUID} or rs_id={GUID}. Our Camera Detail and Server Detail dashboards must query items by tag, not by host name — host names are derived from {#CAM.HW.NAME} via Milestone G3 and aren't predictable from the GUID. Resolution: MilestoneClient::camera($id) uses API::Item()->get(['tags' => [['tag' => 'cam_id', 'value' => $id]]]). Same shape data-bridge.jsx's upstream ActionDashboardData uses for AP fleet items today. Verify Milestone Plan M1's tag scheme lands as documented before our M2 starts.
C8 — Per-source polling-interval mismatch. The dashboards poll every 30 s; XProtect tolerates that for site-level data but not for license details (~3 s per call) or per-camera state (~15 s per call). PacketFence's /api/v1/nodes at 5000 nodes is ~8 s. Resolution: the bridge polls 30 s; the controller decides which upstream calls run on each tick. Cheap (Zabbix item.get) → every tick. Medium (XProtect /sites + /recordingServers) → cached 60 s. Expensive (XProtect /licenseDetails per RS, PF /nodes full) → cached 5 min, refreshed on operator action. Per-call cache key includes hostid/cameraid so per-host views don't share with site-wide.
C9 — {$MILESTONE.SCHEME} means construct URLs, don't assume HTTPS. The macro exists because some installs run XProtect Mgmt over HTTP on a closed VLAN. MilestoneClient reads {$MILESTONE.SCHEME} default https and constructs base URL as {SCHEME}://{HOST}. Don't hardcode https:// — the existing template gets this right, the lifted PHP must too. Add the same SCHEME pattern to PFClient and XIQClient for symmetry (XIQ Cloud is always HTTPS but PF on-prem isn't always).
C10 — XIQ token longevity + refresh. XIQ Cloud bearer tokens are long-lived but rotate on a quarter-ish cadence and on policy changes. Existing XIQClient.php expects a pre-generated token via {$XIQ.TOKEN} macro — no automatic refresh. This is fine for v1.0 — operators rotate quarterly via documented runbook. M5 considers a self-refresh path: store the long-lived OAuth client-credentials grant in macros, mint short-lived tokens server-side. Defer until v2 unless rotation pain shows in M5 testing.
C11 — Polling thundering-herd: 6 bridges × 30 s on a busy dashboard tab = 12 requests/min/operator. An operator with two browser tabs open on different dashboards hits 24/min/operator. At 30 simultaneous operators (busiest TCS shift) that's 720/min into 6 controller actions. Not catastrophic on a tuned Zabbix server — but unnecessary. Resolution: bridges (a) only poll when the tab is visible (visibilitychange event already used by the Camera Wall pattern from Milestone G21); (b) jitter by ±2 s so 30 simultaneous loads don't lock-step; (c) consolidate on one umbrella endpoint (tcs.global.data) where dashboards share data. Pick (a)+(b) for v1.0; (c) is a v2 enhancement if numbers warrant.
C12 — Permissions: secret macros in JSON responses. Zabbix's configuration.export and usermacro.get APIs respect the type=secret flag — secret macros never leak in API responses. But a custom Action*Data.php that echos the macro value into a JSON response (e.g. accidentally surfacing {$MILESTONE.PASSWORD} in a debug field) leaks it. Resolution: code-review checklist for every Action*Data: response body has zero macro values, only resolved-from-macro request results. Unit-test catches by snapshotting one response and grepping for known secret strings. Optional hardening: write a SecretAuditor trait that json_encodes through a recursive scrubber for any string matching a secret-macro pattern.
C13 — SourceBadge propagation across the wire. The dashboards already render <SourceBadge src="zbx|ext|pf|idrac" /> next to every metric. The mock-data files supply the badge inline ({value: 47, src: "zbx"}). Each bridge must propagate the same tag from its controller — controllers know which path produced each value. Resolution: every Action*Data response wraps each metric in {value, src, prev?, history?, missing?} — same shape ActionDashboardData uses for AP items today. The bridge's buildItems() stub-fills src: "zbx" as the default; only deviations (XIQ-only, MS-only, iDRAC-only) need explicit override. Per Milestone Plan G16, conflicting badges on one cell are operationally hazardous — never render two source paths with one badge.
Carry-forward — settled · do not rediscover
AP G21 (carry) — PHP 8.0, not 8.1+. Drop readonly properties, array_is_list(), enums, never, new in initializers, intersection types. Polyfills go in calling namespace because PHP function lookup is namespace-first. Applies to every Action*Data.php, every lib/*Client.php. Source: AP Plan v3.2 §B.
AP G16 (carry) — Module namespace prefix. The existing module's PHP namespace is Modules\TcsDashboard\Actions. New lib/ classes go in Modules\TcsDashboard\Lib — not the source repo's namespaces. Lifting Modules\MilestoneZabbix\Lib\HttpClient verbatim is wrong; rename namespace + class as part of the lift. Source: AP Plan v3.2 §B (G16 collision discovery).
Milestone G1–G15 (carry) — Camera/RS topology + vendor SNMP + ICMP scale. Don't apply to our PHP/JS — they're template-side concerns. But they shape the data we read: cameras and RSes have GUID-keyed items, ICMP runs at {$MILESTONE.CAM.PING.INTERVAL}, vendor sub-templates exist conditionally per sysObjectID. Our bridges read whatever the templates produce; we don't override. Source: Milestone Plan v1.1 §B G1–G15.
Milestone G16 (carry) — SourceBadge correctness. Restated as our C13 above. Promoted to module-level constraint because every bridge must conform — not just the surveillance ones.
Milestone G20 (carry) — No RTSP from Zabbix. Camera Detail dashboard's "stream health" cells (FPS, bitrate, last-frame-age) source from Milestone's view of the stream (per-camera incoming-stream stats from the API or perfcounter on the RS) — never from a Zabbix-initiated RTSP probe. Source: Milestone Plan v1.1 §B G20. Our bridge surfaces only what Milestone reports; if Milestone says "no incoming stream", we render the NO SIGNAL state.
Milestone G21 (carry) — Camera Wall scale. Default OFF, opt-in tweak, 30 s thumbnail interval, pause on tab-hidden. The dashboard already implements this; M2's surveillance bridge respects the tweak — when disabled, the bridge simply doesn't fetch thumbnails (skip the call entirely, don't fetch-and-discard). Source: Milestone Plan v1.1 §B G21.
